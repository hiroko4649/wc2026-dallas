football-data.org API
フェッチ問題と解決の記録

WC2026 Dallas スケジュールアプ — 2026年6月12日

← スケジュールに戻る
概要： ワールドカップ開幕後、試合結果がアプリに表示されず「null - null」と出る問題が発生。 原因はCORSポリシーによりブラウザから外部APIに直接アクセスできないことで、 試行錯誤の末、GitHub Actionsで定期的にAPIを取得してJSONファイルとして リポジトリに保存し、HTMLはそのローカルファイルを読むという構造に変更することで解決した。 ✅ 解決済み
問題発生から解決までの流れ
問題発生
試合結果が「null - null」と表示される
大会開幕後、第1戦（メキシコ vs 南アフリカ）終了後もスコアが表示されず、 代わりに「null - null 終了」と表示された。 「終了」バッジは出ているため、APIからデータは届いているが、 スコアの値だけが取得できていない状態。
第1の試み（失敗）
スコアのフィールド名を修正
football-data.org v4 APIはスコアを score.fullTime.home ではなく score.fullTime.homeTeam で返す場合があると判断し、 両方のフィールド名に対応するよう修正。
// 修正前 var home = score.fullTime.home; // 修正後（両パターンに対応） var h = obj.home != null ? obj.home : (obj.homeTeam != null ? obj.homeTeam : null);
結果：依然として「null - null」のまま。 根本原因はフィールド名ではなかった。
原因判明
CORSエラー — プロキシ経由でAPIキーが通らない
ブラウザのコンソールを確認すると、実際にはAPIへのアクセス自体が失敗していた。
Access to fetch at 'https://api.allorigins.win/raw?url=...' has been blocked by CORS policy: Request header field x-auth-token is not allowed by Access-Control-Allow-Headers in preflight response.
CORS（Cross-Origin Resource Sharing）とは、 ブラウザが異なるドメインへのリクエストを制限するセキュリティ機能。 football-data.orgへの直接アクセスを避けるためプロキシ（allorigins.win等）を 経由していたが、プロキシが X-Auth-Token カスタムヘッダーを転送できないため 認証が通らず、データは取得できていなかった。

つまり「null - null」は「APIが0点を返した」のではなく、 「APIへのアクセス自体が失敗してデータが空」だったのが原因。
第2の試み（失敗）
APIキーをURLパラメータに変更
ヘッダーが通らないなら、URLパラメータとしてトークンを渡す方式に変更。 football-data.orgは ?token=xxx の形式も公式サポートしている。
// ヘッダー方式（失敗） fetch(url, { headers: { 'X-Auth-Token': token } }) // URLパラメータ方式（試み） var apiUrl = 'https://api.football-data.org/v4/...' + '&token=' + API_TOKEN;
結果：3つのプロキシが全て異なる理由で失敗。
allorigins.win → HTTP 522（サーバーダウン） corsproxy.io → HTTP 403（football-data.orgがブロック） codetabs.com → HTTP 400（Bad Request）
football-data.orgがプロキシサーバーのIPアドレスを広くブロックしており、 プロキシ経由のアクセス自体が根本的に機能しない状態だった。
解決策
GitHub Actionsでサーバーサイド取得に切り替え
ブラウザからAPIを叩くのではなく、GitHub Actions（サーバーサイド）が 定期的にAPIを取得して結果をJSONファイルとしてリポジトリに保存。 HTMLはそのJSONファイルを読むだけにする構造に変更。

新構造（✅ 成功）
GitHub Actionsのワークフローファイル（.github/workflows/fetch-matches.yml）：
name: Fetch WC2026 Match Results on: schedule: - cron: '*/10 * * * *' # 10分ごと workflow_dispatch: # 手動実行も可 permissions: contents: write # ← これがないとpush失敗 jobs: fetch: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Fetch matches run: | curl -s \ -H "X-Auth-Token: $TOKEN" \ "https://api.football-data.org/v4/..." \ -o matches.json - name: Commit and push run: | git config user.name "github-actions[bot]" git add matches.json git diff --staged --quiet || \ git commit -m "Update match results" git push
HTMLの読み込みも同じリポジトリの matches.json を参照するよう変更：
// プロキシなし・CORSなし・シンプル fetch('matches.json?_=' + Date.now()) .then(function(res) { return res.json(); }) .then(function(data) { // スコアを表示... });
解決確認
メキシコ vs 南アフリカのスコアが表示された ⚽
GitHub ActionsでAPIの手動実行を行い matches.json が リポジトリに作成されたことを確認。 その後 index.html を更新したことで、 第1戦の結果が正しく表示されるようになった。 以降は10分ごとに自動更新される。

📌 学んだこと

「null - null」はデータが来ていないサイン。 フィールド名の問題より先に、そもそもAPIリクエストが成功しているかをコンソールで確認すべきだった。
CORSはブラウザのセキュリティ機能。 ブラウザから別ドメインへのリクエストには制限がある。 カスタムヘッダー（X-Auth-Token等）を含むリクエストは特に制約が厳しい。
パブリックなCORSプロキシは不安定。 allorigins.win・corsproxy.io等は無料サービスのためダウンタイムがあり、 APIプロバイダーにブロックされることも多い。本番環境では信頼できない。
GitHub ActionsはAPIキーを安全に使える。 サーバーサイドでの実行なのでCORS制限がなく、 定期的なデータ取得と静的ファイル更新の組み合わせは静的サイトの定番パターン。
permissions: contents: writeは必須。 GitHub ActionsがリポジトリにPushするには明示的な書き込み権限の設定が必要。 ないとexit code 128で失敗する。
